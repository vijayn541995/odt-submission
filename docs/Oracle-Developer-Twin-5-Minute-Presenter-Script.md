## Presenter Intent
This is a concise business story, not an engineering deep dive.
Focus on:
- the business problem before coding starts
- how ODT changes that workflow
- why it is different from a generic copilot
- why the solution matters across delivery teams
- why the solution feels safe and scalable

Do not read every bullet on screen. Land the headline, explain the business meaning, and move.

## Slide-by-Slide Talk Track

### Slide 1 - Turning AI Into a Governed Digital Worker for Software Delivery (0:00-0:30)
"Oracle Developer Twin solves a very practical problem. AI can generate code, but teams still lose time in the work before coding starts. ODT turns a ticket into a guided, review-ready delivery flow, while keeping human approval in control."

### Slide 2 - The Expensive Part of Delivery Often Happens Before Code Begins (0:30-0:55)
"Before code starts, teams are already losing time and building risk. People still have to clarify intent, find repo impact, plan tests, and remember accessibility obligations. That hidden layer of work is exactly what ODT makes visible and manageable."

### Slide 3 - ODT Removes Setup Work, Not Developer Judgement (0:55-1:25)
"The point is not to replace developers. The point is to remove setup work. ODT takes intent, turns it into evidence, and prepares execution guidance so developers can focus on judgement instead of repetitive coordination."

### Slide 4 - Why This Is More Than a Coding Copilot (1:25-1:50)
"A generic copilot usually starts with a prompt and returns code or text. ODT starts with a work item, understands the repo, prepares structured outputs, and keeps accessibility, testing, and review in the flow. Codex or Cline can execute. ODT makes execution safer and more consistent."

### Slide 5 - Built. Running. Demoable Today. (1:50-2:20)
"This is a working delivery flow, not a concept mockup. The demo already shows a review-ready run, artifacts on disk, target-repo accessibility verification, and a live worker console. That matters because we are not pitching a future idea. We are showing something that already works."

### Slide 6 - Simple Architecture. Clear Control Points. (2:20-2:50)
"The architecture is intentionally simple. Every layer has one job. ODT plans. Agents execute. Humans decide. That separation is what makes the AI path understandable, safe, and easier to scale."

### Slide 7 - What a Developer Actually Does (2:50-3:25)
"From a developer perspective, the workflow is simple. Set the repo. Run the digital worker. Review the evidence. Delegate if needed. Approve before merge. The engine underneath is rich, but the experience on top stays easy to adopt."

### Slide 8 - Trust Is Built In (3:25-3:55)
"Trust is part of the product, not an extra feature. Human review stays mandatory. Accessibility is visible. Runs create durable evidence. And the OCI path can fall back safely when needed. ODT makes risk visible before it becomes release risk."

### Slide 9 - What This Brings to Oracle (3:55-4:30)
"The bigger value is not only this one demo. ODT is a reusable Oracle pattern for safer, faster software delivery. It improves readiness, raises confidence, makes handoffs easier, and creates a foundation that can grow from frontend today to more stacks and broader digital workers later."

### Slide 10 - From Ticket to Trust (4:30-5:00)
"The takeaway is simple. ODT makes the invisible work before coding visible, guided, and repeatable. This is not just faster AI. It is safer, review-ready delivery. It removes hidden work, keeps ownership human, and is already ready to scale."

## 20-Second Close
"ODT brings structure, explainability, and visible control to the part of software delivery that AI tools usually skip. That is why it is more than a copilot demo. It is a digital worker pattern for real delivery."

## Optional Q&A Backup Lines

- "Why not just use Codex directly?"
  "Because code generation is only one part of the delivery problem. ODT adds intake, repo impact analysis, compliance, explainability, and review structure around execution."

- "What makes this business-relevant?"
  "It improves consistency, makes AI safer to adopt, and creates a reusable enterprise delivery pattern rather than a one-off coding shortcut."

- "How do you reduce AI risk?"
  "Human approval stays mandatory, prompt paths are traceable, accessibility stays in scope, and fallback behavior is built in rather than assumed."

- "What is the long-term opportunity?"
  "Start with frontend SDLC, then extend the same governed-worker model across more stacks and more delivery workflows."
